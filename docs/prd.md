# Product Requirements Document
## Laboratory Information System (LIS)

**Version:** 1.0  
**Status:** Ready for Implementation  
**Stack:** FastAPI (Backend) · PostgreSQL (Database) · React (Frontend)

---

## 1. Product Overview

### 1.1 Purpose

A Laboratory Information System (LIS) that manages the full lifecycle of lab operations — from patient registration and order creation through specimen collection, result entry, verification, report delivery, billing, and doctor commission tracking.

### 1.2 Goals

- Digitize and enforce the end-to-end lab workflow
- Eliminate paper-based result entry and tracking
- Provide role-appropriate interfaces for every actor in the lab
- Enforce clinical validation rules automatically at result entry time
- Produce and deliver patient reports via multiple channels
- Track billing, insurance coverage, and doctor commissions

### 1.3 Non-Goals (v1.0)

- HL7/ASTM instrument integration (instruments table is stubbed, integration deferred)
- Patient portal (patient role is modeled in RBAC but UI deferred)
- Multi-branch support (single lab location)
- Mobile app (web-responsive only)

## 3. System Architecture

The LIS is implemented as a web application with a React frontend, a FastAPI
backend, PostgreSQL persistence, and background processing for long-running or
side-effect-heavy workflows.

### 3.1 Frontend

- React single-page application served separately from the API
- Uses the OpenAPI-generated TypeScript client for all backend calls
- Stores access tokens in memory only and relies on the backend auth flow defined
  in section 5.2
- Enforces permission-aware navigation and action visibility using the current
  user's effective roles and permissions
- Performs client-side form validation for immediate feedback, while treating API
  validation as the source of truth

### 3.2 Backend API

- FastAPI application exposed under `/api/v1`
- Routers handle transport concerns only:
  - request parsing and Pydantic validation
  - authentication dependencies
  - RBAC permission checks
  - response model serialization
  - conversion of service errors into the standard API error shape
- OpenAPI is the contract between backend and frontend; route tags and operation
  IDs must remain stable enough for generated client usage
- All authenticated requests resolve the current user through FastAPI
  dependencies before reaching workflow code

### 3.3 Service Layer

Business workflows live in service modules rather than routers. Services own
transaction boundaries, state transitions, cross-table orchestration, and
business rule enforcement.

Core services include:

- `order_service` — order creation, item expansion, specimen assignment, invoice
  creation, and commission entry creation
- `specimen_service` — specimen determination, collection, rejection, and status
  transitions
- `billing_service` — invoice creation, payment tracking, voiding, and refunds
- `commission_service` — commission configuration lookup, entry creation, and
  payout batching
- `result_service` — result entry, edits, status transitions, and verification
- `validation_service` — reference range matching, abnormal/critical flags,
  absurd range blocking, and delta checks
- `consistency_service` — cross-analyte rule evaluation
- `reflex_service` — reflex rule evaluation and reflex order item creation
- `report_service` — PDF generation, report versioning, release, and voiding
- `notification_service` — critical, report, order, and delivery notifications
- `audit_service` — request-aware audit event creation

Clinical workflows that update multiple tables must run inside a single database
transaction. If validation or a business rule fails, all related changes are
rolled back.

### 3.4 Persistence

- PostgreSQL is the system of record
- Alembic migrations define and evolve the database schema
- Primary keys are UUIDs generated by PostgreSQL
- Database names use `snake_case`
- Timestamps are stored as `TIMESTAMPTZ` in UTC
- Mutable operational records use `is_deleted` soft deletes where appropriate
- Immutable clinical, billing, report, and audit records use voiding,
  replacement versions, or append-only history instead of physical deletion
- Database constraints enforce invariants that must never be bypassed, including
  uniqueness, foreign keys, partial unique indexes, and exclusion constraints
- Audit logs capture inserts, updates, and deletes with old/new JSON values and
  the user who performed the action

### 3.5 Background Processing

Background workers handle work that should not block the user request once the
database transaction has safely committed:

- report PDF generation and object storage upload
- notification delivery over email, SMS, WhatsApp, and in-app channels
- delivery status updates and retryable outbound integrations
- validation, consistency, and reflex processing when a workflow is explicitly
  configured to run asynchronously

Synchronous clinical validation remains inside the request transaction whenever
the result must be blocked before save.

### 3.6 File Storage

- Generated report PDFs are stored outside the database in object storage
- The database stores report metadata, version, delivery status, and storage URL
- Report downloads use short-lived pre-signed URLs, with a maximum expiry of 15
  minutes

## 5. API Design

### 5.1 Conventions

All API endpoints are exposed under `/api/v1`.

General conventions:

- Request and response bodies are JSON unless an endpoint explicitly returns a
  file or binary stream
- URL paths use `kebab-case`
- JSON fields use `snake_case`
- Path parameters use UUIDs for persisted resources
- Collection endpoints use plural nouns, for example `/patients`
- Item endpoints use `/resources/{id}`
- State-changing actions that are not simple CRUD use action subresources, for
  example `/orders/{id}/cancel`, `/results/{id}/verify`, and
  `/reports/{id}/void`
- `DELETE` means soft delete unless the resource is immutable; immutable
  clinical, billing, report, and audit records must be voided, versioned, or
  rejected through explicit action endpoints instead

List conventions:

- List endpoints support server-side pagination with `skip` and `limit`
- Search uses `search` for free-text matching
- Filters use explicit query parameters, for example `status`, `doctor_id`,
  `patient_id`, `date_from`, and `date_to`
- Sorting uses `sort_by` and `sort_order`, where `sort_order` is `asc` or `desc`
- List responses use:
```json
{
  "data": [],
  "count": 0
}
```

Authentication and authorization:

- The current v1 auth strategy keeps the existing FastAPI template routes in
  section 5.2
- Authenticated endpoints require a bearer access token
- Access tokens are stored in frontend memory only
- Refresh/session behavior should follow the existing backend auth implementation
  until a future auth cleanup is explicitly planned
- Every protected endpoint checks RBAC permissions in the API dependency layer
  before calling service code

OpenAPI conventions:

- Every router must have a stable tag matching its resource area
- Operation IDs must remain stable because the frontend client is generated from
  OpenAPI
- Request and response schemas must be explicit Pydantic models
- Public response models must not expose password hashes, internal tokens, or
  implementation-only fields
- Errors use the standard response shape defined in section 10

## 6. Core Feature Modules

---

### 6.1 Patient Registration

**Actors:** Receptionist

**Behaviour:**
- Search patients by name, identifier, or date of birth before creating new
- Create patient with demographics
- Attach one or more insurance policies; mark one as primary
- Editing is soft — all changes captured in `AuditLogs`

**Frontend:**
- Patient search with debounced lookup
- Patient form with validation
- Insurance sub-form with primary toggle enforced (one primary at a time)

**Backend:**
- `GET /patients?search=` with trigram index on `FirstName`, `LastName`, `Identifier`
- Enforce one primary insurance per patient via partial unique index

---

### 6.2 Order Management

**Actors:** Receptionist, Lab Manager

**Behaviour:**
- Create order linked to patient, doctor, optional insurance policy, and patient context
- Add tests (items) or panels from catalog
- Panels are auto-expanded into constituent `OrderItems`
- System auto-determines required specimens from `CatalogSpecimenRequirements`, deduplicates by type, creates `OrderSpecimens`, and links each `OrderItem` to its `OrderSpecimen`
- System auto-generates invoice on order save
- System auto-creates `DoctorCommissionEntry` on order save using active `DoctorCommissionConfig`
- Order can be edited (tests added/removed) before specimen collection
- Order can be cancelled from `registered` or `collected` status only

**Frontend:**
- Order form with patient/doctor lookup
- Catalog picker with search, category filter, item/panel toggle
- Panel expansion preview before adding
- Specimen requirements summary shown before confirming order
- Invoice preview on order save

**Backend services:**
- `order_service.create_order()` — orchestrates items, specimens, invoice, commission in a single DB transaction
- `specimen_service.determine_specimens()` — deduplicates specimen types across all order items
- `billing_service.create_invoice()` — sums `PriceCharged` across items
- `commission_service.create_entry()` — looks up active config and creates entry

---

### 6.3 Specimen Collection

**Actors:** Phlebotomist

**Behaviour:**
- Phlebotomist opens collection queue — all orders in `registered` status
- Views required tubes per order (type, color, volume, instructions)
- Records collection: sets `CollectionTime`, `CollectedBy`, `Status = collected`
- Order status transitions to `collected`
- Specimen rejection: sets `Status = rejected`, `RejectionReasonId`, marks linked `AnalyteResults` as rejected

**Frontend:**
- Collection queue list, grouped by order
- Tube checklist per order with color-coded specimen type indicators
- Reject button with reason picker

---

### 6.4 Result Entry

**Actors:** Technician

**Behaviour:**
- Technician sees queue of orders with collected specimens not yet fully resulted
- Enters result values per analyte
- On save, system automatically:
  1. Finds best-matching `ValidationRule` (by analyte, gender, age, context, priority)
  2. Checks absurd range → blocks if violated
  3. Checks panic range → sets `IsCritical = true`, triggers `CriticalNotification`
  4. Checks normal range → sets `IsAbnormal = true`
  5. Runs delta check against last verified result for same patient+analyte
  6. Runs `ConsistencyRules` for all analytes in the `OrderItem` — blocks on error, warns on warning
  7. Evaluates `ReflexRules` → auto-adds reflex tests as new `OrderItems` if triggered
  8. Evaluates `CalculationFormula` for calculated analytes
- Result status set to `resulted`
- Order status transitions to `in_progress` on first result, `partial_results` if some done

**Frontend:**
- Result entry screen grouped by order then by test
- Analyte input fields typed by `ResultType` (numeric input, text area, dropdown for options, image upload)
- Inline flags shown after save: ABNORMAL, CRITICAL, DELTA, warnings
- Consistency rule violations shown inline with severity color coding
- Reflex test auto-add notification with confirmation

**Backend services:**
- `result_service.enter_results()` — persists results, fires validation pipeline
- `validation_service.validate()` — rule matching + flag setting
- `consistency_service.check()` — formula evaluation
- `reflex_service.evaluate()` — trigger detection + order item creation
- All run within a single transaction; validation errors roll back the result

---

### 6.5 Critical Notifications

**Actors:** Technician (creator), Supervisor / Pathologist (acknowledger)

**Behaviour:**
- Auto-created when `IsCritical = true` on a result
- Must be acknowledged before result can be verified
- Notification method recorded (call, SMS, in-app, email)
- Acknowledgement timestamp and actor recorded

**Frontend:**
- Critical notifications badge in nav (unacknowledged count)
- Notification list with result detail, patient info, notified-to user
- Acknowledge button with method picker

---

### 6.6 Result Verification

**Actors:** Pathologist

**Behaviour:**
- Pathologist sees queue of resulted but unverified `OrderItems`
- Reviews all analyte values, flags, and comments
- Can edit a result before verifying (correction)
- Can add comments per analyte result
- Verifies each result → `Status = verified`, `VerifiedById`, `VerifiedAt`
- Cannot verify if unacknowledged critical notifications exist for that result
- When all analytes of an order are verified → `Order.Status = completed`
- Partial verification → `Order.Status = partial_results`

**Frontend:**
- Verification queue grouped by order
- Result card per analyte showing value, flags, reference range, delta
- Edit inline before verifying
- Comment thread per result
- Verify all / verify one actions

---

### 6.7 Report Generation & Release

**Actors:** Pathologist

**Behaviour:**
- Triggered after order reaches `completed` status
- Pathologist selects template (or default is used)
- System compiles all verified results and generates PDF → stores to S3
- Report record created with `Version = 1`
- Delivered via selected channel: print, email, WhatsApp, portal
- `DeliveryStatus` tracked per report row
- Re-release: voids current version, creates `Version = n+1` with corrected results
- All report releases logged in `Notifications`

**Frontend:**
- Report generation button on completed orders
- Template picker
- Channel selector (multi-select for email + print simultaneously)
- Report history per order showing version, channel, delivery status
- Void + re-release flow with confirmation

**Backend:**
- `report_service.generate()` — PDF generation (WeasyPrint or equivalent)
- `report_service.deliver()` — channel-specific dispatch (background task)
- `notification_service.log()` — creates `Notifications` row per delivery

---

### 6.8 Billing & Invoicing

**Actors:** Receptionist, Finance, Lab Manager

**Behaviour:**
- Invoice auto-created on order save (Version 1)
- Supports partial payment tracking (`AmountPaid`, `PaymentStatus`)
- Invoice can be voided and reissued (`Version = n+1`, old `IsVoided = true`)
- Refunds tracked via `payments:refund`
- Insurance-covered items use `InsurancePricing` rate instead of catalog price
- Payment method recorded per transaction

**Frontend:**
- Invoice view on order detail
- Payment collection form with method picker and amount
- Void + reissue flow
- Invoice version history

---

### 6.9 Doctor Commissions

**Actors:** Finance, Lab Manager

**Behaviour:**
- Commission entry auto-created on order save using active `DoctorCommissionConfig`
- Separate rates for insurance vs non-insurance orders
- Config enforces non-overlapping date ranges per doctor
- Finance reviews pending entries per doctor
- Creates payout batch (`DoctorCommissionPayment`) linking selected entries
- Entries marked `paid` on payout

**Frontend:**
- Commission entries list filterable by doctor, date range, payout status
- Payout batch creation with entry selection and total preview
- Payout history per doctor
- Commission config management per doctor

---

### 6.10 Catalog & Rules Management

**Actors:** Lab Manager, Super Admin

**Behaviour:**

**Catalog:**
- Create/edit tests (items) with analytes, specimen requirements, pricing
- Create/edit panels with constituent tests and sort order
- Categories with sort order for display grouping
- Insurance pricing overrides per provider per catalog item

**Validation Rules:**
- Per analyte, per gender, per age range, per patient context
- Priority-based: most specific rule wins
- Absurd, panic, normal range configuration
- Delta check threshold

**Consistency Rules:**
- Formula-based cross-analyte checks
- Warning or error severity
- Plain-English description required

**Reflex Rules:**
- Trigger analyte + operator + value
- Action: auto-add catalog item to order

**Frontend:**
- Catalog list with type filter (item/panel), category filter, search
- Test builder: analyte assignment, specimen requirements, pricing tabs
- Panel builder: constituent test picker with drag-and-drop sort order
- Validation rule matrix per analyte
- Consistency rule builder with formula input and analyte tag picker
- Reflex rule builder

---

### 6.11 User & Role Management

**Actors:** Lab Manager, Super Admin

**Behaviour:**
- Create users, assign roles
- Roles carry permissions via `RolePermissions`
- Users can have multiple roles
- Role assignment can have `ExpiresAt` for temporary grants
- Deactivating a user (`IsActive = false`) blocks login
- All role changes captured in `AuditLogs`

**Frontend:**
- User list with role badges
- User form with role multi-select
- Role list with permission matrix view
- Role editor: permission toggle grid by resource × action
- Temporary role grant with date picker

---

### 6.12 Audit Log

**Actors:** Lab Manager, Super Admin

**Behaviour:**
- Every insert, update, delete on every table captured automatically
- Old and new values stored as JSON
- Filterable by table, action, user, date range

**Backend:**
- Implemented as SQLAlchemy event listeners on session flush
- Captures `performed_by_id` from request context

**Frontend:**
- Audit log table with filters
- Row expand to show old/new JSON diff

## 8. Key Business Rules

| Rule | Enforcement |
|------|-------------|
| Panel items must be Type=item | DB check constraint |
| Panel cannot reference itself | Application layer |
| Catalog.Code and Analytes.Code must be unique | DB unique index |
| One primary insurance per patient | Partial unique index |
| No overlapping commission configs per doctor | PostgreSQL exclusion constraint (tsrange) |
| Result cannot be verified with unacknowledged critical notifications | Application layer |
| Order can only be cancelled from registered or collected | Application layer state machine |
| Invoice void requires creating a new version | Application layer |
| Report re-release voids previous version | Application layer |
| ValidationRule priority: highest wins | Application layer (ORDER BY priority DESC LIMIT 1) |
| Reflex-added tests are marked IsReflexAdded=true | Application layer |
| Commission entry auto-created on order save | Application layer (order_service) |

## 9. Data Validation (API Layer)

All request bodies validated with Pydantic v2:

- Required vs optional fields enforced
- Enum fields: validated against defined values only
- Numeric ranges: min/max where applicable
- Dates: ISO 8601 format
- UUIDs: validated on path parameters
- `ResultValue` validated against `ResultType` (numeric string for numeric analytes, etc.)

## 10. Error Handling

| Code | Meaning |
|------|---------|
| 400 | Validation error (Pydantic) or business rule violation |
| 401 | Unauthenticated |
| 403 | Insufficient permission (RBAC guard) |
| 404 | Resource not found |
| 409 | Conflict (e.g. duplicate code, overlapping config) |
| 422 | Unprocessable entity |
| 500 | Internal server error |

All errors return:
```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Human readable message",
    "details": {}
  }
}
```

## 14. Security Requirements

- All endpoints require authentication except the current auth/password recovery
  endpoints listed in section 5.2
- Refresh/session behavior follows the existing backend auth implementation until
  a future auth cleanup is explicitly planned
- Access token stored in memory only (not localStorage)
- Rate limiting on `/login/access-token` (5 attempts / 15 min)
- All DB queries parameterized (SQLAlchemy ORM — no raw string interpolation)
- Audit log cannot be modified or deleted by any role
- S3 report URLs are pre-signed with short expiry (15 min)
